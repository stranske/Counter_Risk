# UI Subsystem Audit

Summary: The user-facing surfaces (tkinter GUI, CLI, web_demo, chat) are generally well-structured, typed, and testable, with launch logic deliberately split into pure helpers. The most significant real risks for local Windows operators are (1) the GUI runs the entire pipeline synchronously on the Tk main thread, freezing the window for the duration of a run, and (2) the GUI "discover" mode routes into an interactive `input()` stdin prompt that has no console in a windowed PyInstaller `.exe`. Several "Open ..." buttons can also silently no-op on bad date input. Duplication is concentrated in `runner_launch.py` (four near-identical open-* functions) and in the parallel command-building logic that exists in both `gui/runner.py` and `runner_launch.py`.

## Code Quality Findings

### [BLOCKER] GUI "discover" mode calls `input()` on stdin — no console in the windowed .exe
- `src/counter_risk/gui/runner.py:149-150` builds `--discover` when discovery mode is "discover" (selectable in the readonly combobox at `runner.py:360-366`).
- The GUI runner callback (`cli/__init__.py:236-237`, `_gui_command`) re-enters `main`, which routes `--discover` to `_run_with_discovery` (`cli/__init__.py:211-212, 260`).
- `_run_with_discovery` calls `resolve_discovery_selections(result)` with the default prompt (`cli/__init__.py:293`), which when an input has multiple matches calls `_prompt_user_selection_stdin` → `input(...)` (`io/discover.py:242`).
- In a GUI-launched/PyInstaller windowed build there is no stdin; `input()` raises `EOFError`, which `_prompt_user_selection_stdin` converts to `raise SystemExit(1)` (`io/discover.py:243-244`). `SystemExit` is not caught by `_run`'s `except Exception` (`runner.py:316`, `SystemExit` derives from `BaseException`), so the entire Tk app exits with no operator-facing message whenever discovery finds multiple candidates. Recommendation: have the GUI pass a Tk-based `prompt_fn` (a selection dialog) into discovery, or disable/guard the "discover" option in the windowed GUI and route multi-match resolution through the UI.

### [MAJOR] Full pipeline runs synchronously on the Tk main thread (UI freeze, no cancel)
- `src/counter_risk/gui/runner.py:292-321` `_run` calls `execute_gui_run(...)` directly in the button callback, which runs the entire pipeline (`execute_gui_run` → `runner(cli_args)` → `main` → `run_pipeline_with_config`). The only concession is `status_var.set("Running..."); root.update_idletasks()` at `runner.py:296-297`.
- During a multi-minute run the window is unresponsive (no repaint, no cancel, appears "hung" to the OS). For a non-technical month-end operator this is a notable usability gap. Recommendation: run the pipeline on a worker thread and marshal results back via `root.after(...)`, or at minimum show an explicit modal "running" state and disable the buttons.

### [MAJOR] GUI "Open ..." buttons can silently no-op on invalid date input
- `_open_output`, `_open_manifest`, `_open_summary`, `_open_ppt` (`runner.py:323-339`) call `_resolve_existing_output_dir(current)` with no try/except. That path runs `parse_as_of_month` → `date.fromisoformat` (`runner_launch.py:87`), which raises `ValueError` on a malformed As-Of Date.
- Unlike `_run` (which wraps everything and shows a `messagebox`), these handlers have no error handling, so on bad input the Tk callback raises, prints a traceback to a (nonexistent in .exe) console, and the button appears to do nothing. Recommendation: wrap these handlers the same way `_run` is wrapped, surfacing a `messagebox.showerror`. Also `_open_path` returns `False` when the path does not exist (`runner.py:204-205`) but no caller surfaces that to the operator — a missing folder click is indistinguishable from success.

### [MINOR] `parse_as_of_month` validation runs twice per GUI run
- `execute_gui_run` calls `parse_as_of_month(state.as_of_date)` at `runner.py:175` purely for validation, then `_build_cli_args` parses it again at `runner.py:138`. Harmless but redundant; the comment at `runner.py:174` documents the intent, so this is polish only.

### [MINOR] `web_demo.run_web_demo_pipeline` permanently mutates `os.environ`
- `src/counter_risk/web_demo.py:198` sets `os.environ[_OFFLINE_ENV_VAR] = "1"` and never restores it. For a one-shot demo process this is fine, but if the helper is ever called in-process (e.g. tests, a host app), it leaks offline mode into the rest of the process. Recommendation: set it within a context manager / restore prior value in a `finally`.

### [MINOR] `_default_runner` / handler do not surface pipeline `print` output in the windowed GUI
- The GUI callback ultimately calls `main`, whose handlers emit progress and error detail via `print` (e.g. `cli/__init__.py:374` "Counter Risk run failed: {exc}"). In the windowed `.exe` there is no stdout, so the only operator signal is the GUI's `result_var`/`messagebox`. The GUI does map exit codes to "Exit code N" (`runner.py:313`) but the human-readable failure reason printed by the CLI is lost. The richer translator `map_runner_error_to_operator_message`/`format_launch_error_for_runner` (`runner_launch.py:46-83`) exists but is not wired into the tkinter GUI's error display. Recommendation: capture stderr/stdout or the exception text and route it through `format_launch_error_for_runner` for the GUI.

### Positive: CLI and chat error handling is solid
- `cli/__init__.py` consistently returns deterministic exit codes (2 for input/settings errors, 1 for run failures) and wraps `_run_workflow_mode` in a broad try/except with a user-facing message (`cli/__init__.py:373-375`). Settings loading distinguishes OSError vs JSONDecodeError with clear messages (`cli/__init__.py:395-401`).
- `chat/ui.py:52-68` cleanly separates expected validation errors (`ChatSessionError`, `PromptInjectionError`) returned to the user from unexpected `RuntimeError` (logged via `.exception`, generic message returned) — good operator-facing hygiene. Input is validated (empty message, provider/model support) before any provider call.
- `chat/context.py` consistently maps low-level IO/parse failures to a single `RunContextError` with actionable text (e.g. parquet/pandas guidance at `context.py:197-199`).

## Duplication / Simplification Opportunities

### [MAJOR] Four near-identical `open_*` functions in `runner_launch.py`
- `open_output_folder` (`runner_launch.py:259-304`), `open_data_quality_summary` (`307-352`), `open_manifest` (`355-395`), and `open_ppt_output_folder` (`398-440`) share the same shape: resolve path → exists check (error_code N) → try open (error_code N+1) → nonzero exit-code branch → success `LaunchStatus`. Only the path resolver, the "not found" message, and the error_code offsets differ. This is ~180 lines that condense to one helper parametrized by `(resolve_path_fn, not_found_message, base_error_offset, opener)` without behavior change. The four public wrappers can remain thin shims to preserve the existing API/error codes.

### [MAJOR] Command/CLI-arg construction duplicated across `gui/runner.py` and `runner_launch.py`
- `runner_launch.py` builds CLI command *strings* (`build_command`, `build_discovery_dry_run_command`, `build_discovery_run_command`, `runner_launch.py:214-256`) for the Excel/`.bas` path, while `gui/runner.py:_build_cli_args` (`runner.py:131-163`) builds the same `run --config ... --as-of-date/--as-of-month ... --output-dir ... --settings ...` invocation as an argv *list* for the tkinter path. The run-mode→config-path mapping is also duplicated: `_RUN_MODE_TO_CONFIG` (`runner.py:27-31`, forward slashes) vs `resolve_config_path` (`runner_launch.py:176-181`, backslashes). The two will drift as flags evolve. Recommendation: centralize argument assembly in one place that both the string builder and the GUI argv builder consume.

### [MINOR] Settings payload builders duplicated
- `gui/runner.py:_build_settings_payload` (`runner.py:97-105`) and `runner_launch.py:build_runner_settings_payload` (`runner_launch.py:190-205`) produce the same JSON object (same five keys, `sort_keys=True`). They differ only in `.strip() or default` vs plain `.strip()`. Consolidate to one builder.

### [MINOR] Repeated `last_output_dir or _resolve_existing_output_dir(current)` in four GUI handlers
- `runner.py:325, 329, 334, 339` repeat the same fallback expression. Extract a small `_current_output_dir()` helper; this also gives a single place to add the missing error handling noted above.

### [MINOR] Path-normalization round-trips in GUI resolvers
- `_resolve_output_dir` / `_resolve_existing_output_dir` (`runner.py:71-90`) wrap backslash-producing helpers and then `.replace("\\", "/")` back to POSIX, passing a `directory_exists` lambda that itself re-replaces separators. The `runner_launch` helpers are Windows-`\`-centric (`normalize_path_separators`, `runner_launch.py:95-99`) while the GUI is cross-platform; the back-and-forth is a smell worth a dedicated cross-platform path helper rather than string surgery at each call site.

## Notable Strengths

- Launch decision logic is deliberately extracted into pure, dependency-injected functions (`runner_launch.py`, `runner_date_control.py`, `gui/runner.py:execute_gui_run` takes `runner`/`directory_exists`/`temp_dir` callbacks), making the operator surfaces unit-testable without Excel or a display. `launch_gui` accepts an injected `runner` and `initial_state`.
- The GUI degrades gracefully when tkinter is unavailable, with an actionable message pointing at `counter-risk run` / `gui --headless` (`runner.py:255-257`), and a true `--headless` path exists for CI (`cli/__init__.py:239-254`).
- The chat layer has a thorough, well-documented prompt-injection defense (trusted/untrusted delimiters, boundary-token validation, spreadsheet-vector and zero-width/bidi detection, formula-prefix neutralization) in `chat/session.py:400-563`, with the honest caveat comment at `session.py:400` that it does not replace model-side safety.
- `web_demo.py` enforces a strong no-egress contract: it asserts every fixture source lives under `tests/fixtures` (`web_demo.py:100-113`) and verifies the LangChain chat runtime is not newly imported during a demo run (`web_demo.py:123-133`), comparing against pre-run state to avoid false positives under xdist.
- Argument parsing supports operator-friendly aliases (`--as-of-date`/`--as-of-month`) and explicit tri-state `--export-pdf`/`--no-export-pdf` (`cli/__init__.py:65-97`), and `mapping_diff_report` correctly rejects the mutually exclusive `--manifest-json`/`--run-dir` combination with exit code 2 (`cli/mapping_diff_report.py:148-150`).
